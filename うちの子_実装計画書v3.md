## 1. プロダクト概要

### 1.1 テーマとの接続（小さくなる日本）

| 軸 | 意味 |
|----|------|
| **作品との距離を小さく** | 欲しいグッズが手に入らない問題を、データ×3Dプリンターで解決する |
| **小さいコミュニティを大きくする** | マイナーキャラや同人作品のファンが、物理のグッズで繋がれる場を作る |
| **ネットの力を使ってモノの世界を小さくする** | データは1秒で届く。3Dデータ×ローカル印刷で送料・納期・流通の壁を消す |

### 1.2 コンセプト

> 「うちの子」を、自分の手で立体にする。

写真・イラスト1枚から3Dメッシュを生成し、3Dプリンターで印刷できるSTLデータとして出力するWebサービス。生成したデータはコミュニティで共有・販売でき、公式もデータを出品できる新しいグッズ流通プラットフォーム。

### 1.3 解決するエピソード

| エピソード | 深層の課題 |
|-----------|-----------|
| マイナーなキャラを推しているのでグッズが見つからない | **数が少ない** → 公式が作らない規模のキャラはグッズが存在しない |
| グッズが手に入らないくらい人気のキャラを推している | **数が多い** → 人気すぎて買えない・転売される |
| 今は売っていないグッズが欲しい | **すでに終わっている** → 廃盤・絶版 |
| 売り出たばかりのキャラのグッズが欲しい | **まだ作られていない** → 流行の先取り |
| グッズのないようなものが好き（名言・子どもの絵・友人のイラスト） | **そもそも商品化されない** → 個人の思い出・非公式創作 |

**共通する構造的バグ：** 「欲しい」と「買える」の間に、製造・流通・在庫の巨大なコストの壁がある。データ+ローカル印刷でこの壁を消す。

### 1.4 解決できるもの

- **物流問題**：データを送るだけ。送料ゼロ。
- **企業側の新しい収入源**：公式が3Dデータを出品して収益化できる。
- **「欲しい」需要を満たす**：マイナー・廃盤・新作・個人制作すべてに対応。

## 2. サービス設計（3つの利用フロー）

### フロー① 写真・イラストから3Dデータを生成する

```
ユーザーが写真/イラストを1枚アップロード
  → AIが3Dモデルを生成（Tripo3D / Wonder3D）
  → STLをダウンロード → 3Dプリンターで印刷
```

**ターゲット：** 個人ファン、同人作家、子どもの絵を立体にしたい親

### フロー② 好きな3Dデータを見つけてダウンロード・購入する

```
コミュニティから3Dデータを検索・閲覧
  → データをダウンロード（無料）または購入（有料）
  → 自宅またはサービス経由で3Dプリント
```

**ターゲット：** 3Dプリンターを持っているユーザー

### フロー③ 公式が3Dデータをアップして値段をつけて販売する

```
メーカー・作家・公式アカウントが3Dデータをアップロード
  → 価格・ライセンスを設定して出品
  → ファンがデータを購入 → 印刷
```

**ターゲット：** キャラクター版権を持つ企業・個人作家・同人サークル

## 3. 技術スタック

### 3.1 全体構成

```
[ブラウザ]
  Firebase Studio で開発した React + Vite アプリ
  Firebase Hosting でホスティング
       ↓ API呼び出し
[APIサーバー]
  Railway にデプロイした FastAPI (Python)
  ・Tripo3D API 連携（実写→3D）
  ・Wonder3D / HuggingFace 連携（イラスト→3D）
  ・Depth Anything V2（即時プレビュー）
  ・trimesh（GLB→STL変換）
       ↓ データ保存
[データベース]
  Neon（PostgreSQL）
  ・ユーザー情報
  ・作品メタデータ
  ・いいね・購入履歴
       ↓ ファイル保存
[ストレージ]
  Firebase Storage
  ・STL / GLB ファイル
  ・サムネイル画像
```

### 3.2 写真→3D変換：技術比較と採用方針

| 技術 | 入力 | 品質 | 速度 | 採用 |
|------|------|------|------|------|
| **Tripo3D API** | 1枚〜 | ★★★★☆ | 1〜3分 | ✅ 実写・コスプレ向け |
| **Wonder3D / CRM** | 1枚 | ★★★★★ | 3〜8分 | ✅ アニメ・イラスト向け |
| Depth Anything V2 | 1枚 | ★★☆☆☆ | 即時 | 即時プレビュー専用（STL不可） |
| Gaussian Splatting | 20〜100枚 | ★★★★★ | 数十分 | デモ用見本の事前作成のみ |
| NeRF / LiDAR | 専用機器 | ★★★★★ | 数時間〜 | Phase 2以降 |

**採用方針：** Tripo3D（実写）+ Wonder3D（イラスト）の2本立て。アップ直後は Depth Anything V2 で即時プレビュー表示し、バックで本格メッシュ生成。

### 3.3 フロントエンド

| カテゴリ | 採用技術 | 理由 |
|---------|---------|------|
| フレームワーク | React + Vite | Firebase Studioとの相性◎ |

| スタイリング | Tailwind CSS | 高速開発。AI補完と相性◎ |
| 状態管理 | Zustand | 軽量で2週間開発に適切 |

### 3.4 バックエンド

## 4. データベース設計（Neon / PostgreSQL）

### 4.1 テーブル設計

#### `users` テーブル

```sql
CREATE TABLE users (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  firebase_uid TEXT UNIQUE NOT NULL,
  display_name TEXT,
  is_creator   BOOLEAN DEFAULT FALSE,  -- 出品者フラグ
  has_printer  BOOLEAN DEFAULT FALSE,  -- 3Dプリンター保有フラグ
  created_at   TIMESTAMPTZ DEFAULT NOW()
);
```

#### `works` テーブル

```sql
CREATE TABLE works (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID REFERENCES users(id) ON DELETE CASCADE,
  title         TEXT NOT NULL,
  genre         TEXT,  -- figure/anime/cosplay/original/official/other
  is_official   BOOLEAN DEFAULT FALSE,
  price         INTEGER DEFAULT 0,  -- 0=無料, 単位:円
  thumbnail_url TEXT,
  stl_url       TEXT,
  glb_url       TEXT,
  task_id       TEXT,  -- Tripo3D / Wonder3D のジョブID
  status        TEXT DEFAULT 'pending',  -- pending/processing/done/failed
  likes         INTEGER DEFAULT 0,
  downloads     INTEGER DEFAULT 0,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);
```

#### `likes` テーブル

```sql
CREATE TABLE likes (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    UUID REFERENCES users(id) ON DELETE CASCADE,
  work_id    UUID REFERENCES works(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, work_id)
);
```

#### `purchases` テーブル（フロー③）

```sql
CREATE TABLE purchases (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    UUID REFERENCES users(id),
  work_id    UUID REFERENCES works(id),
  amount     INTEGER NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 4.2 インデックス

```sql
CREATE INDEX idx_works_user_id  ON works(user_id);
CREATE INDEX idx_works_genre    ON works(genre);
CREATE INDEX idx_works_status   ON works(status);
CREATE INDEX idx_likes_work_id  ON likes(work_id);
```

## 5. API設計（FastAPI）

### 5.1 エンドポイント一覧

| Method | Path | 認証 | 説明 |
|--------|------|------|------|
| GET | `/health` | 不要 | ヘルスチェック |
| POST | `/api/depth` | 不要 | 画像→Depthマップ推定（即時プレビュー用） |
| POST | `/api/generate` | 必要 | 画像→3D変換ジョブ開始、task_idを返す |
| GET | `/api/task/{task_id}` | 必要 | ジョブのステータス・進捗をポーリング |
| POST | `/api/convert/stl` | 必要 | GLB/OBJ → STL変換 |
| POST | `/api/works` | 必要 | 作品をNeonに保存 |
| GET | `/api/works` | 不要 | 作品一覧取得（フィルタ・ページネーション） |
| GET | `/api/works/{id}` | 不要 | 作品詳細取得 |
| POST | `/api/works/{id}/like` | 必要 | いいね追加・解除 |

### 5.2 認証方式

Firebase Auth の IDトークンをヘッダーで受け取り、FastAPI 側で検証する。

```
Authorization: Bearer {Firebase IDトークン}
```

FastAPI 側で `firebase-admin` SDK を使って検証：

```python
from firebase_admin import auth

async def verify_token(authorization: str = Header(...)):
    token = authorization.replace("Bearer ", "")
    decoded = auth.verify_id_token(token)
    return decoded["uid"]
```

### 5.3 外部API利用フロー

**実写写真（Tripo3D）**

```
POST /api/generate
  → Tripo3D: /v2/openapi/upload（画像アップロード）
  → Tripo3D: /v2/openapi/task（ジョブ作成）→ task_id返却
  → フロント: GET /api/task/{task_id} を3秒ごとにポーリング
  → 完了: STLダウンロードURL → Firebase Storage保存 → Neonに保存
```

**イラスト・アニメ絵（Wonder3D）**

```
POST /api/generate?mode=anime
  → gradio_client → HuggingFace Spaces（Wonder3D）呼び出し
  → OBJファイル取得 → trimeshでSTL変換 → Firebase Storage保存
```
